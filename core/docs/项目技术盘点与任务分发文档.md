# 项目技术盘点与任务分发文档

> 项目：煤炭双轨定价多尺度预测系统  
> 版本：v1.0（用于并行开发派工）  
> 目标：将现有“可运行原型”升级为“可答辩、可复现、可扩展的工程化系统”

---

## 1. 现状与差距分析（Gap Analysis）

### 1.1 已具备能力（当前可用）

#### 1) 技术栈与工程骨架
- **后端**：Flask（`app.py`），提供 `dashboard`、`predict` 等 API。
- **前端**：Vue3 + ECharts（单页驾驶舱），已有双轨图、指标卡、Tab 图表等可视化框架。
- **模型层（已落地）**：
  - 日度：`LSTM + Transformer`（`LSTMTransformerRegressor`）
  - 月度：`LightGBM`
  - 年度：`SVR + Ridge` 混合（`YearlyBundle`）
  - 合同价映射：`ContractPriceMapper`（线性映射 + 规则走廊）
- **特征工程**：多尺度聚合与自动特征构建（滚动、滞后、交互等），并有特征筛选流程。
- **评估与报告**：已有 `MAPE / RMSE / MAE`、滚动回测结果与 metadata 产物。
- **部署链路**：GitHub Actions -> Hugging Face Space 自动同步部署。

#### 2) 数据层现状
- 已有结构化数据与文本/天气数据输入管线。
- 近期新增了对 `预测数据.xlsx` 的容错读取（自动识别日期/数值列，空值插值，JSON 安全返回）。

---

### 1.2 与“最终目标”的核心差距

#### A. 数据一致性与异构对齐仍不充分
- 外部异构数据（Wind/Excel/文本）虽可接入，但**统一主键、频率、单位、口径治理**未形成标准化层。
- 日/月/年数据虽然能聚合，但**缺失值策略与异常值策略尚未形成统一规则文档与可追溯日志**。

#### B. 模型族谱不完整（与理论方案仍有差距）
- 当前主链路仅覆盖 `LSTM-Transformer + LightGBM + SVR/Ridge`。
- 理论规划中常见的对照与增强分支（如 **GRU、Transformer-only、Prophet/XGBoost 月度分支**）尚未工程化并行接入。
- **图神经网络（GNN）尚未构建**：尚无“政策-供需-价格”关系图、图特征构造、图模型训练与线上推理链路。

#### C. 可解释性与策略闭环不足
- 目前有指标展示，但缺少完整的“因果路径/特征贡献/事件影响”解释链。
- 前端虽然有事件打桩，但“事件来源-指数变化-预测偏移量”尚未形成标准可解释报告。

#### D. 工程稳健性与交付规范待加强
- 前端图表在隐藏容器初始化、JSON 严格解析、缓存一致性等问题已出现过，需要系统化测试治理。
- 缺少端到端回归测试、接口契约测试与“发布前质量闸门”。

---

### 1.3 最终目标定义（本轮冲刺）

- 构建“**数据可信 + 模型可比 + 结果可解释 + 前端可演示 + 部署可回滚**”的竞赛级系统：
  1. 数据层：异构源统一标准化与质量审计；
  2. 模型层：完成主链路稳定 + 对照分支 + GNN 原型；
  3. 解释层：形成事件驱动解释闭环；
  4. 平台层：前后端稳定运行 + 自动化发布与监控。

---

## 2. 任务拆解与派发（Task Breakdown）

> 以下任务按四位成员并行分工。每个模块包含：任务目标、技术要求、交付物、验收标准。  
> 建议按照 “M1 -> M2 -> M3 -> M4” 有依赖地并行推进。

---

### 模块 A：数据治理与异构对齐（负责人：数据工程）

#### 任务目标
- 建立统一的数据标准层，打通结构化数据、Wind 导出数据、Excel 临时数据、文本与天气数据。
- 形成可复用的数据清洗、对齐、质检和落盘流程，保证模型输入稳定。

#### 技术要求
- 建立统一字段字典：`字段名 / 含义 / 单位 / 频率 / 来源 / 缺失策略`。
- 统一主键和时间粒度：至少保证 `date` 日频可对齐；月频/年频要有明确映射规则。
- 实施数据质量规则：
  - 缺失率阈值与插值策略（线性/前向填充/禁填）；
  - 异常值检测（IQR 或 z-score）；
  - 重复记录与口径冲突检测；
  - 质量报告自动输出（含告警等级）。
- 输出标准化中间层（建议 `data/curated/` 分层保存 `daily/monthly/yearly`）。

#### 交付物
- `docs/data_dictionary.md`
- `src/data_standardize.py`（或同等模块）
- `reports/data_quality_audit_latest.json`（自动生成）
- 一套可复现的数据快照（含原始->清洗->对齐日志）

#### 验收标准
- 任意一次全量构建后，模型训练输入不再因字段缺失/类型漂移失败。
- 关键字段缺失率、插值率、异常率可追踪且可视化输出。

---

### 模块 B：NLP 与事件驱动因子引擎（负责人：NLP工程）

#### 任务目标
- 将政策文本、新闻舆情、极端天气事件结构化为可学习因子，并提供“事件->指标->价格影响”的解释链路。

#### 技术要求
- 保持现有 `BERT + LDA` 框架，同时补齐工程规范：
  - 事件提取（政策发布时间、关键词、情绪方向、强度值）；
  - 日度聚合规则（窗口、加权、衰减）；
  - 因子稳定性分析（漂移、稀疏、突变）。
- 建立“事件打桩数据接口”：
  - 统一输出字段：`date, event_type, title, impact_score, sentiment_score, source_url`。
- 与前端联动：保证 hover 可展示可解释文本。

#### 交付物
- `src/nlp_event_engine.py`（或同等模块）
- `data/curated/event_markers.csv`
- `reports/nlp_factor_drift_report.json`
- `docs/nlp_factor_spec.md`

#### 验收标准
- 事件打桩数据可稳定输出且无空结构崩溃。
- 至少给出 3 类典型事件（政策/舆情/天气）可在前端主图正确展示。

---

### 模块 C：多尺度模型增强与GNN原型（负责人：算法工程）

#### 任务目标
- 在现有多尺度主链路上，补齐对照模型与图模型原型，形成“可比较、可解释、可汇报”的模型体系。

#### 技术要求
- 在不破坏当前主链路前提下增加实验分支：
  - 日度对照：GRU 或 Transformer-only（至少 1 个）
  - 月度对照：Prophet/XGBoost（二选一或都实现）
  - 年度：现有 SVR+Ridge 保留，补充参数搜索记录规范
- **GNN 原型（重点）**：
  - 构建关系图：政策-供需-价格-情绪节点；
  - 定义边关系（相关性/时滞因果先验）；
  - 完成最小可运行图模型（如 GAT/GraphSAGE）；
  - 输出对比指标（MAPE/RMSE）与消融实验。
- 预测稳健性：
  - 保留日度限幅逻辑；
  - 输出“原始预测 vs 约束后预测”对照。

#### 交付物
- `src/models_experimental.py`（或扩展至现有 `src/models.py`）
- `reports/model_benchmark_matrix.json`（多模型对照）
- `reports/gnn_prototype_eval.json`
- `docs/model_benchmark_and_gnn.md`

#### 验收标准
- 主链路不回归（现有线上接口可用）。
- 新增至少 2 个对照分支 + 1 个 GNN 原型，且有统一评估结果。

---

### 模块 D：平台工程、前端驾驶舱与发布稳定性（负责人：全栈/MLOps）

#### 任务目标
- 将系统提升到“可稳定演示”的交付水平：前端稳、接口稳、发布稳、故障可回滚。

#### 技术要求
- 前端工程化：
  - 图表懒加载与 tab 可见时 resize；
  - 全局数值格式化（3 位小数）；
  - 异常态统一（空数据、接口超时、JSON 解析失败）。
- 后端接口契约治理：
  - 严格 JSON（NaN/Inf -> null）；
  - 响应 schema 固定化；
  - 加入健康检查与版本标识。
- CI/CD 质量闸门：
  - 加入基本测试（接口 contract test + 前端关键渲染 smoke test）；
  - GitHub Actions 失败自动告警；
  - 预留回滚策略（上一个稳定 commit 一键回退）。

#### 交付物
- `docs/api_contract.md`
- `tests/test_api_contract.py`（至少覆盖 dashboard/predict）
- `tests/test_frontend_smoke.md`（或自动化脚本）
- `docs/release_runbook.md`（发布/回滚手册）

#### 验收标准
- Space 连续部署 5 次无中断可访问；
- 前端关键模块（主图、月度、年度、在线预测）在冷启动后可正常渲染；
- 任一接口异常时页面不黑屏，能给出可读错误提示。

## 3. 模块间接口与依赖约定（执行关键）

### 3.1 统一接口字段（最低要求）
- `dashboard_data.market_layer.timeline / market_price / contract_price`
- `dashboard_data.nlp_layer.policy_strength / sentiment_score`
- `prediction.next_day_market_price / next_day_contract_price / next_month_market_price / next_year_market_price`
- `excel_overlay.timeline / price / null_ratio / points`

### 3.2 交付节奏建议（两周冲刺）
- **第 1-3 天**：模块 A 完成字段字典与标准化基线；模块 D 固化接口契约；
- **第 4-7 天**：模块 B 完成事件引擎；模块 C 完成对照模型分支；
- **第 8-10 天**：模块 C 完成 GNN 原型；模块 D 完成前端联调；
- **第 11-14 天**：全链路压测、答辩材料与演示彩排。

## 4. 风险清单与应对

- **风险1：异构数据口径冲突导致模型漂移**  
  处置：字段字典强制审查 + 数据质量门禁。

- **风险2：前端可视化依赖数据完整，空值导致模块空白**  
  处置：后端统一 sanitize + 前端默认值与降级渲染。

- **风险3：新增模型分支影响线上稳定**  
  处置：主链路与实验链路隔离，灰度切换。

- **风险4：部署链路偶发失败影响展示**  
  处置：发布前预热 + 回滚脚本 + 备用演示快照。

- 每日站会按“交付物清单”检查完成度；
- 任何需求变更必须先更新本文件中的接口与验收标准。
