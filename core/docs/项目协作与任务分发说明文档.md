# 《项目协作与任务分发说明文档》

## 项目背景（一句话版）

**项目名称与目标：** 本项目为面向煤炭市场的**双轨定价多尺度价格预测系统**（日/月/年尺度 + 长协/现货映射），通过 Flask 在线服务与 Hugging Face Space 对外提供预测与可视化仪表盘。  
**最终目标：** 在工业级数据契约与可复现流水线之上，完成从异构数据接入、特征与风控计算、多模型融合到线上监控与回测报告的全链路闭环，并可选接入 C++ 高性能核心以支撑规模化与低延迟推理。

## 2. 差距分析与现状盘点（Gap Analysis）

### 2.1 当前已具备的能力

- **服务与展示**：`core/app.py` + `templates` + `static` 提供仪表盘与 `/api/dashboard_full` 等 CSR 数据接口；根目录与 Docker/Procfile/CI 已形成可部署链路。  
- **建模与回测**：`core/src/models.py`、`backtest.py`、`pipeline.py`、`train.py` 支撑多尺度训练与回测摘要。  
- **配置与契约**：`core/configs/`、`core/schemas/` 已搭好分域配置与 JSON Schema 框架。  
- **C++ 与 pybind**：`cpp_core/` 具备目录与 CMake/绑定骨架，便于高性能扩展。  
- **模块化 Python**：`core/python/` 下 CLI 与 `serving` 子包为从单体 `app.py` 迁出预留位置。

### 2.2 与“最终目标”的主要差距

| 类别 | 现状 | 差距 / 空壳说明 |
|------|------|-----------------|
| **数据分层** | 文档中规划 `data/raw`、`data/clean`、`data/mart`，仓库以 `.gitignore` 忽略运行时数据目录 | **物理目录与稳定样例数据集**未在仓库中标准化落地；ingestion 与 mart 层缺少统一落盘与版本化规范。 |
| **配置驱动** | YAML 分散在 `configs/` | **主训练/服务路径未全部改为读 YAML**，与 `core/app.py` 仍存在硬编码或环境变量混用，需统一“单一配置源”。 |
| **C++ 核心** | 头文件与 cpp 骨架存在 | **业务逻辑多为占位**；Python 侧未形成稳定的 pybind 调用链与性能基准测试。 |
| **`core/python/` 子包** | 大量 `README.md` 占位 | **与 `core/src/` 实际代码未完全迁移或重复**，工业化入口（`run_*`）与 `core/app.py` 需明确唯一真相来源。 |
| **测试与 CI** | HF 同步工作流存在 | **单元测试、契约测试、schema 校验 CI**未在文档树中强制化；合并前质量门禁不足。 |
| **观测与运维** | 基本 Flask 日志 | **结构化日志、指标、健康检查、模型热更新策略**可加强，便于四人并行时排障。 |

### 2.3 后续需优先补充的“基础设施”

1. **数据契约强制执行**：在 ingestion 出口与训练入口对 `schemas/*.json` 做校验（失败即拒绝或隔离脏数据）。  
2. **最小可复现数据集**：`data/clean` 的匿名样例或合成数据 + README 说明字段含义，避免成员本地路径不一致。  
3. **分支保护与 PR 模板**：见第 4 节，把代码审查与测试要求固化到流程。  
4. **接口版本约定**：`/api/dashboard_full` 字段变更时同步 `prediction.schema.json` 与前端 `dashboard.js`，避免静默破坏。

## 4. GitHub 团队协作规范（Git Workflow）

### 4.1 分支命名

| 类型 | 命名示例 | 说明 |
|------|----------|------|
| 功能 | `feature/data-ingestion-wind` | 新功能开发 |
| 修复 | `fix/api-json-nan` | Bug 修复 |
| 文档 | `docs/collab-readme` | 纯文档 |
| 发布 | `release/v1.2.0` | 发布准备（可选） |

主分支：**`main`** 为受保护分支；个人不直接推送，一律经 PR 合并。

### 4.2 工作流程（建议）

1. 从最新 `main` 拉取：`git pull origin main`。  
2. 创建功能分支：`git checkout -b feature/<简短描述>`。  
3. 小步提交：提交信息使用约定前缀，如 `feat:`、`fix:`、`docs:`、`refactor:`、`chore:`。  
4. 推送远程：`git push -u origin feature/<简短描述>`。  
5. 在 GitHub 打开 **Pull Request**：填写模板（见下）、指定审查者（至少一名非作者）。  
6. **CI/检查通过**且 **至少一人 Approve** 后，由维护者或作者执行 **Squash merge** 或 **Merge commit**（团队统一一种）。  
7. 合并后删除远程功能分支，本地同步 `main`。

### 4.3 Pull Request 要求

- **标题**：清晰动词 + 范围，例如：`feat(core): add schema validation to ingestion output`。  
- **描述**：动机、主要改动、如何本地验证（命令）、是否影响 API/数据库/配置。  
- **关联**：关联 Issue 编号（若有）。  
- **审查关注点**：数据契约、向后兼容、密钥与大数据未误入仓库。

### 4.4 合并前审核清单（审查者）

- [ ] 变更范围与 PR 描述一致，无无关大重构。  
- [ ] 若改 API 或 JSON 结构：`schemas/` 与前端/调用方已同步。  
- [ ] 新增依赖已写入 `core/requirements.txt` 并说明用途。  
- [ ] 大文件、密钥、本地路径未提交。  
- [ ] 文档或 `README` 中命令仍指向 `core/` 唯一入口。

## 5. 团队协作注意事项（Collaboration Notes）

### 5.1 接口与数据格式

- **JSON 字段命名**：统一使用 **snake_case**（与当前 Python/Flask 习惯一致）；前端展示层再转中文标签。  
- **数值**：后端避免向 JSON 输出 `NaN`/`Inf`（已在服务端 sanitize）；前端金额类统一 **保留小数位规则**（如三位），与 `dashboard.js` 中 `formatPrecision` 一致。  
- **日期**：统一 **ISO 8601 字符串**（如 `YYYY-MM-DD`）或 Unix 毫秒二选一，全项目统一并在 `contracts.yaml` 注明。

### 5.2 配置与环境

- **单一真相**：业务参数优先落在 `core/configs/`，环境差异用 `dev/staging/prod` 区分；禁止在代码中写死可被配置覆盖的路径。  
- **Python 版本**：与 `Dockerfile`/`runtime.txt` 一致（当前为 3.11 系），本地建议使用 `python3.11` + `venv`。  
- **依赖变更**：任何 `pip` 包增减必须更新 `core/requirements.txt` 并在 PR 说明。

### 5.3 模块边界与对接

- **数据工程** 输出 Clean 层 → **算法** 只从 Clean/Mart 约定路径读取，不直接读 Raw。  
- **算法** 变更模型输出维度 → 必须通知 **全栈** 更新 `prediction.schema.json` 与 `dashboard.js`。  
- **C++** 输出特征列名 → 与 `feature_frame.schema.json` 一致，并在 PR 中附列名对照表。

### 5.4 文件与仓库卫生

- **大文件**：`*.xlsx`、PDF、模型权重等按 `.gitignore` 策略处理；若需共享样例，使用脱敏小文件或网盘链接（README 说明）。  
- **并行编辑**：`core/app.py` 与 `core/static/js/dashboard.js` 冲突概率高，合并前优先拉取 `main`；大改动前在 Issue 或群聊占位说明。

### 5.5 沟通与里程碑

- **短期对齐**：每周固定同步接口字段与配置变更（可贴到 Issue 或 `core/docs/` 变更日志）。  
- **发布前**：角色 C 执行部署分支冒烟；角色 B 提供回测摘要基准；角色 A 确认数据版本号；角色 D 确认 C++ 可选路径关闭时不影响默认 Python 流程。
